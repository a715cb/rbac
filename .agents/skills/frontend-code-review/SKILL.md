# 前端代码审查

## 意图

当用户要求审查前端代码（尤其是 `.tsx`、`.ts` 或 `.js` 文件）时使用此技能。支持两种审查模式：

1. **待提交变更审查** — 检查准备提交的已暂存/工作区文件，在提交前标记检查清单违规
2. **指定文件审查** — 审查用户指定的特定文件，报告相关检查清单发现

严格遵循下方检查清单，适用于每个适用文件和模式。

## 检查清单

参见 [references/code-quality.md](references/code-quality.md)、[references/performance.md](references/performance.md)、[references/business-logic.md](references/business-logic.md) 获取按类别划分的活检查清单——将其视为要遵循的规则规范集。

为每个规则违规标记紧急度元数据，以便后续审查者可以优先修复。

## 审查流程

1. 打开相关组件/模块。收集与类名、React Flow Hooks、属性记忆化和样式相关的行。
2. 对于审查点中的每条规则，记录代码偏离的位置并捕获代表性代码片段。
3. 按以下模板组织审查部分。先按**紧急**标记分组，然后按类别顺序（代码质量、性能、业务逻辑）排列。

## 必要输出

调用时，响应必须严格遵循以下两种模板之一：

### 模板 A（有发现）

```
# 代码审查

发现 <N> 个紧急问题需要修复：

## 1 <问题简要描述>
文件路径: <path> 行 <line>
<相关代码片段或指针>

### 建议修复
<建议修复的简要描述>

---

...（每个紧急问题重复）...

发现 <M> 条改进建议：

## 1 <建议简要描述>
文件路径: <path> 行 <line>
<相关代码片段或指针>

### 建议修复
<建议修复的简要描述>

---

...（每条建议重复）...
```

如果没有紧急问题，省略该部分。如果没有改进建议，省略该部分。

如果问题数量超过10个，总结为"10+ 个紧急问题"或"10+ 条改进建议"，仅输出前10个问题。

不要压缩各部分之间的空行；保持原样以确保可读性。

如果使用模板A（即有问题需要修复）且至少有一个问题需要代码更改，在结构化输出后附加一个简短的后续问题，询问用户是否希望应用建议的修复。例如："是否希望我使用建议修复部分来解决这些问题？"

### 模板 B（无问题）

```
## 代码审查

未发现问题。
```

## 使用场景

- **审查待提交的变更**：在准备提交代码（git commit）前，审查所有已修改或已暂存的文件，提前识别并标记不符合规范的代码，避免潜在的Bug或性能问题被合入代码库。
- **审查指定的文件**：当需要对特定文件或模块进行重构、优化或问题排查时，将相关文件交由此技能审查，快速获取质量评估结果及改进建议。
- **获取结构化的修复报告**：代码审查发现问题后，自动整理并输出标准化报告，按紧急程度排序，标注文件路径、行号、代码片段，并给出可执行的修复方案。
