# Pull Request创建器

此技能引导创建遵循仓库标准的高质量Pull Request。

## 工作流程

按照以下步骤创建Pull Request：

1. **分支管理**：检查当前分支，避免直接在 `main` 上工作。
   - 运行 `git branch --show-current`。
   - 如果当前分支是 `main`，创建并切换到一个新的描述性分支：
     ```bash
     git checkout -b <new-branch-name>
     ```

2. **定位模板**：在仓库中搜索Pull Request模板。
   - 检查 `.github/pull_request_template.md`
   - 检查 `.github/PULL_REQUEST_TEMPLATE.md`
   - 如果存在多个模板（例如在 `.github/PULL_REQUEST_TEMPLATE/` 中），询问用户使用哪一个，或根据上下文选择最合适的一个（例如 `bug_fix.md` vs `feature.md`）。

3. **阅读模板**：读取已识别的模板文件内容。

4. **起草描述**：创建严格遵循模板结构的PR描述。
   - **标题**：保留模板中的所有标题。
   - **检查清单**：审查每个项目。已完成的标记为 `[x]`。不适用的项目保持未勾选或标记为 `[ ]`（取决于模板的说明），或在模板允许灵活性的情况下移除（但优先保持未勾选以确保透明度）。
   - **内容**：用清晰、简洁的变更摘要填写各部分。
   - **相关问题**：链接此PR修复或相关的任何问题（例如"Fixes #123"）。

5. **预检**：在创建PR之前，运行工作区预检脚本，确保所有构建、lint和测试检查通过。
   ```bash
   npm run preflight
   ```
   如果任何检查失败，在继续创建PR之前解决问题。

6. **创建PR**：使用 `gh` CLI创建PR。为避免多行Markdown的shell转义问题，先将描述写入临时文件。
   ```bash
   # 1. 将起草的描述写入临时文件
   # 2. 使用 --body-file 标志创建PR
   gh pr create --title "type(scope): 简洁描述" --body-file <temp_file_path>
   # 3. 删除临时文件
   rm <temp_file_path>
   ```
   - **标题**：如果仓库使用[约定式提交](https://www.conventionalcommits.org/)格式，确保标题遵循该格式（例如 `feat(ui): 添加新按钮`、`fix(core): 修复崩溃`）。

## 原则

- **合规性**：绝不忽略PR模板。它的存在是有原因的。
- **完整性**：填写所有相关部分。
- **准确性**：不要为未完成的任务勾选复选框。

## 使用场景

- **一键创建符合规范的PR**：在本地完成新功能开发或Bug修复并已提交代码后，调用此技能让AI自动执行分支检查、查找并应用PR模板、运行预检脚本，并生成标题和描述都完全符合项目规范的PR。
- **引导贡献者完成首次代码提交**：新团队成员或外部贡献者不熟悉项目的提交流程和规范时，AI以智能向导形式自动完成模板查找、脚本执行等繁琐操作，仅需用户填写必要的标题与描述。
- **自动执行创建PR前的质量检查**：在正式创建PR之前，AI自动运行项目预设的预检脚本，执行所有必要的构建、单元测试和代码风格检查。如果任何检查失败，AI会中止提交流程并提示开发者进行修复。
