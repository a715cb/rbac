# 代码审查员

此技能引导智能体对本地开发和远程Pull Request进行专业且全面的代码审查。

## 工作流程

### 1. 确定审查目标

* **远程PR**：如果用户提供了PR编号或URL（例如"审查PR #123"），则以该远程PR为目标。
* **本地变更**：如果没有提及特定PR，或用户要求"审查我的代码"，则以当前本地文件系统状态（已暂存和未暂存的变更）为目标。

### 2. 准备工作

#### 远程PR：

1. **检出**：使用GitHub CLI检出PR。
   ```bash
   gh pr checkout <PR_NUMBER>
   ```
2. **预检**：执行项目的标准验证套件，尽早捕获自动化失败。
   ```bash
   npm run preflight
   ```
3. **上下文**：阅读PR描述和任何现有评论，以理解目标和历史。

#### 本地变更：

1. **识别变更**：
   * 检查状态：`git status`
   * 读取差异：`git diff`（工作区）和/或 `git diff --staged`（已暂存）
2. **预检（可选）**：如果变更较大，询问用户是否要在审查前运行 `npm run preflight`。

### 3. 深度分析

基于以下支柱分析代码变更：

* **正确性**：代码是否在不引入Bug或逻辑错误的情况下实现了其既定目的？
* **可维护性**：代码是否整洁、结构良好、易于理解和修改？考虑代码清晰度、模块化和对既有设计模式的遵循等因素。
* **可读性**：代码是否有适当的注释（必要时）并按照项目编码风格指南一致地格式化？
* **效率**：变更是否引入了明显的性能瓶颈或资源低效？
* **安全性**：是否存在潜在的安全漏洞或不安全的编码实践？
* **边界情况与错误处理**：代码是否适当处理了边界情况和潜在错误？
* **可测试性**：新增或修改的代码是否有充分的测试覆盖（即使预检通过）？建议可提高覆盖率或鲁棒性的额外测试用例。

### 4. 提供反馈

#### 结构

* **概述**：审查的高层次概览。
* **发现**：
  * **关键问题**：Bug、安全问题或破坏性变更。
  * **改进建议**：更好的代码质量或性能建议。
  * **小问题**：格式或次要风格问题（可选）。
* **结论**：明确的建议（批准 / 要求修改）。

#### 语气

* 建设性、专业、友好。
* 解释*为什么*需要修改。
* 对于批准，承认贡献的具体价值。

### 5. 清理（仅远程PR）

* 审查完成后，询问用户是否要切换回默认分支（例如 `main` 或 `master`）。

## 使用场景

- **审查远程PR**：完成功能开发或问题修复并提交PR后，提供PR编号或URL，AI会自动检出代码、运行预检脚本、阅读PR描述与评论，对代码开展深度分析并给出反馈。
- **审查本地代码变更**：在提交代码或创建PR前，对本地修改进行审查。AI通过git status、git diff等命令检查工作区中的代码改动，进行分析并反馈。
- **提供深度分析与结构化反馈**：从多维度开展代码质量分析，涵盖正确性、可维护性、可读性与执行效率、安全性与测试完整性等维度，以结构化形式输出反馈。
