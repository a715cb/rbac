# 8. 权限体系

## 8.1 RBAC 权限模型

系统采用 **用户 → 角色 → 权限** 的经典 RBAC 模型，权限分为三个维度：

```
用户(User) ──多对多──> 角色(Role) ──多对多──> 菜单(Menu)
                                    ├──多对多──> 按钮(MenuButton)
                                    └──多对多──> 接口(Api)
```

**权限检查流程**:

```
请求 → AuthCheck 中间件 (JWT验证)
     → ApiPermission 中间件 (接口权限检查)
         ├─ 超级管理员 → 直接放行
         ├─ 白名单路径 → 直接放行
         ├─ 接口未配置 → 默认放行
         └─ 接口已配置 → 检查用户是否拥有该接口权限码
```

## 8.2 数据权限

数据权限通过角色的 `data_scope` 字段控制，在查询数据时自动过滤：

| data\_scope | 名称     | 说明         | 过滤逻辑                      |
| ----------- | ------ | ---------- | ------------------------- |
| 1           | 全部数据   | 可查看所有数据    | 无过滤                       |
| 2           | 本部门数据  | 仅查看所属部门    | `dept_id IN (用户部门ID列表)`   |
| 3           | 本部门及以下 | 查看所属部门及子部门 | `dept_id IN (部门及子部门ID列表)` |
| 4           | 仅本人数据  | 只能查看自己的数据  | `id = 当前用户ID`             |
| 5           | 自定义    | 指定部门范围     | `dept_id IN (自定义部门ID列表)`  |

## 8.3 认证流程

```
┌────────┐    POST /admin/login     ┌──────────┐
│  登录页 │ ──────────────────────> │  后端API  │
│        │ <────────────────────── │          │
│        │  {access_token,         │  验证密码  │
│        │   refresh_token,        │  生成JWT  │
│        │   user_info}            │  记录日志  │
└────────┘                         └──────────┘
     │
     │ 存储 Token 到 SessionStorage
     │ 获取用户信息 (menus, permissions, roles)
     │ 动态生成路由
     │
┌─────▼──────────────────────────────────────────┐
│                 已认证状态                        │
│                                                  │
│  每次请求:                                       │
│  1. Axios 拦截器自动添加 Authorization Header    │
│  2. 401 响应 → 自动使用 Refresh Token 刷新       │
│  3. 刷新失败 → 跳转登录页                        │
│  4. Token 过期 → 跳转登录页                      │
└──────────────────────────────────────────────────┘
```
