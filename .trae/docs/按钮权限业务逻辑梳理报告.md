## 1. 权限控制流程图

### 1.1 整体权限验证流程

```
用户请求
  │
  ├── 前端路由守卫（beforeEach）
  │     │
  │     ├── 检查路由 meta.roles / meta.permissions
  │     │     └── 调用 hasAnyRoles() / hasAnyPermission()
  │     │           ├── super_admin → 直接放行
  │     │           └── 非超管 → 检查权限码列表
  │     │
  │     └── 无权限 → 跳转 403 页面
  │
  ├── 前端按钮级控制（v-auth / hasPermission）
  │     │
  │     ├── v-auth="'code'"     → 无权限时隐藏元素
  │     ├── v-auth-disabled     → 无权限时禁用元素
  │     └── hasPermission()     → 编程式判断
  │           ├── super_admin → true
  │           └── 非超管 → permissions.includes(code)
  │
  └── 后端 API 请求验证
        │
        ├── AuthCheck 中间件 → JWT 身份认证
        │
        ├── ApiPermission 中间件 → 接口权限验证
        │     ├── super_admin → 直接放行
        │     ├── 白名单接口 → 直接放行
        │     ├── 未配置接口 → 默认放行（记录日志）
        │     └── 已配置接口 → 验证 API 权限码
        │
        └── 控制器层 → 数据权限过滤
              ├── super_admin → 不受部门限制
              └── 非超管 → getScopedDeptIds() 过滤
```

## 2. 超级管理员与非超级管理员权限差异分析

### 2.1 权限范围对比

| 维度 | 超级管理员 (super_admin) | 非超级管理员 |
|------|------------------------|-------------|
| **菜单权限** | 拥有所有启用菜单的访问权限，无需角色-菜单关联 | 仅拥有角色-菜单关联（sys_role_menu）中配置的菜单 |
| **按钮权限** | 拥有所有启用按钮的操作权限，自动获取全表按钮码 | 仅拥有角色-按钮关联（sys_role_menu_button）中配置的按钮 |
| **API权限** | 拥有所有启用接口的访问权限，自动获取全表API码 | 仅拥有角色-API关联（sys_role_api）中配置的接口 |
| **数据权限** | 不受部门限制，可查看所有数据（getScopedDeptIds 返回空数组） | 受 data_scope 级别约束（1-5级数据权限） |
| **路由守卫** | 所有路由均放行 | 需检查路由 meta 中的 roles/permissions |

### 2.2 操作限制对比

| 操作类型 | 超级管理员 | 非超级管理员 |
|---------|----------|-------------|
| **新增操作** | 所有模块均可新增 | 仅拥有对应 `:add` 权限码的模块可新增 |
| **编辑操作** | 所有模块均可编辑 | 仅拥有对应 `:edit` 权限码的模块可编辑 |
| **删除操作** | 所有模块均可删除 | 仅拥有对应 `:delete` 权限码的模块可删除 |
| **状态切换** | 所有 Switch 开关可用 | 仅拥有对应 `:status` 权限码的开关可用 |
| **批量操作** | 所有批量按钮可用 | 仅拥有对应 `:batch_*` 权限码的按钮可用 |
| **权限分配** | 可为任何角色分配权限 | 需拥有 `:permission` 权限码 |
| **数据范围** | 不受部门数据过滤 | 受 data_scope 约束，可能只能看到本部门/本人数据 |

### 2.3 功能可见性对比

| 功能区域 | 超级管理员 | 非超级管理员 |
|---------|----------|-------------|
| **导航菜单** | 显示所有启用菜单 | 仅显示角色关联的菜单 |
| **工具栏按钮** | 显示所有操作按钮 | 仅显示有权限码的按钮（无权限按钮被隐藏） |
| **操作列按钮** | 显示所有行操作 | 仅显示有权限码的操作（无权限操作被隐藏） |
| **Switch 开关** | 所有状态开关可操作 | 无权限时开关 disabled（灰色不可点击） |
| **角色权限分配弹窗** | 可打开并操作 | 需拥有 `permission_role:permission` 权限码 |
| **数据范围配置弹窗** | 可打开并操作 | 需拥有 `permission_role:data_scope` 权限码 |

### 2.4 判断逻辑差异伪代码

```
// 后端判断（AdminAuth.php）
function checkPermission(permission, type):
    if userId <= 0: return false
    if isSuperAdmin(): return true          // 超管直接放行
    if type == 1: return permission in userPermissions
    if type == 2: return permission in menuCodes
    if type == 3: return permission in apiCodes

function isSuperAdmin():
    superAdminCode = Config.get('auth.super_admin_code', 'super_admin')
    return superAdminCode in userRoleCodes

function getButtonCodes():
    if isSuperAdmin(): return getAllButtonCodes()  // 超管返回全表
    return queryByRoleButtonAssociations()         // 非超管按角色查询

// 前端判断（user.ts）
function hasPermission(permission):
    if roleCodes.includes('super_admin'): return true  // 超管直接放行
    return permissions.includes(permission)

function hasRole(roleCode):
    if roleCodes.includes('super_admin'): return true
    return roleCodes.includes(roleCode)
```

## 4. 权限机制详解

### 4.1 权限判断条件

#### 4.1.1 角色标识判断

| 判断位置 | 判断方式 | 配置来源 |
|---------|---------|---------|
| 后端 AdminAuth::isSuperAdmin() | `in_array(Config::get('auth.super_admin_code'), userRoleCodes)` | `config/auth.php` → `super_admin_code` |
| 前端 UserStore | `roleCodes.value.includes('super_admin')` | 硬编码 `'super_admin'` |
| 后端 ApiPermission 中间件 | `$auth->isSuperAdmin()` | 同 AdminAuth |

**配置文件**：`backend/config/auth.php`

```php
'super_admin_code' => env('AUTH_SUPER_ADMIN_CODE', 'super_admin'),
```

**注意**：前端硬编码了 `'super_admin'` 字符串，未从后端配置同步，若修改后端 `super_admin_code` 配置，前端判断将不一致。

#### 4.1.2 用户组归属判断

系统通过 `sys_user_role`（用户-角色多对多关联）确定用户归属的角色组：

```
用户 → sys_user_role → 角色(sys_role) → sys_role_menu_button → 按钮权限码
```

一个用户可拥有多个角色，按钮权限取所有角色的**并集**。

#### 4.1.3 数据范围限制

| data_scope | 名称 | 过滤逻辑 | 超管行为 |
|-----------|------|---------|---------|
| 1 | 全部数据 | 不添加部门过滤 | 同（返回空数组） |
| 2 | 本部门数据 | WHERE dept_id IN (用户部门ID列表) | 不受限制 |
| 3 | 本部门及以下 | WHERE dept_id IN (用户部门+子部门ID) | 不受限制 |
| 4 | 仅本人 | WHERE user_id = 当前用户ID | 不受限制 |
| 5 | 自定义 | WHERE dept_id IN (data_scope_dept_ids) | 不受限制 |

**多角色数据权限取最宽松值**：`resolveDataScope()` 取所有角色中 `data_scope` 值最小的（权限范围最大的）。

### 4.2 权限继承关系

```
超级管理员 (super_admin)
  └── 继承方式：硬编码绕过，不通过关联表
  └── 所有菜单、按钮、API权限均直接获取全表数据
  └── 数据权限不受部门限制

管理员 (admin, role_id=2)
  └── 继承方式：通过 sys_role_menu_button 关联
  └── 初始化时拥有所有按钮权限（SELECT 全表插入）
  └── 后续新增按钮需手动分配

人事经理 (hr_manager, role_id=4)
  └── 继承方式：通过 sys_role_menu_button 关联
  └── 仅拥有用户管理(7个) + 部门管理(3个) = 10个按钮权限

普通用户 (user, role_id=3)
  └── 继承方式：通过 sys_role_menu_button 关联
  └── 仅拥有日志删除(2个) + 日志导出(1个) = 3个按钮权限
```

**权限继承规则**：
1. 超级管理员不通过关联表继承，而是代码层面直接放行
2. 非超级管理员的权限通过角色-按钮关联表精确控制
3. 多角色用户的按钮权限取所有角色的**并集**
4. 角色间无层级继承关系，每个角色的权限独立配置

### 4.3 权限动态调整机制

#### 4.3.1 权限分配方式

| 操作 | API | 说明 |
|------|-----|------|
| 分配按钮权限 | `POST /admin/roles/:id/buttons` | 全量替换模式：先删除旧关联，再插入新关联 |
| 分配菜单权限 | `POST /admin/roles/:id/menus` | 全量替换模式 |
| 分配API权限 | `POST /admin/roles/:id/apis` | 全量替换模式 |
| 设置数据范围 | `PUT /admin/roles/:id/data-scope` | 更新角色的 data_scope 和 data_scope_dept_ids |

#### 4.3.2 缓存刷新机制

权限变更后自动清理缓存，确保实时生效：

| 触发操作 | 清理范围 | 清理方法 |
|---------|---------|---------|
| 角色-按钮关联变更 | 该角色关联的所有用户权限缓存 | `clearRoleCache(roleId)` → `clearAllUserCache()` |
| 菜单/按钮增删改 | 全局缓存 + 所有用户缓存 | `clearGlobalCache()` + `clearAllUserCache()` |
| 用户角色变更 | 该用户权限缓存 | `clearCache(userId)` |
| 用户登出 | 该用户权限缓存 | `clearAllCache()` |

#### 4.3.3 临时权限与有效期

**当前系统不支持**临时权限分配和权限有效期管理。所有权限分配均为持久化配置，需手动调整。

## 5. 权限判断伪代码

### 5.1 前端按钮权限判断

```typescript
// 权限判断入口（v-auth 指令内部调用）
function checkButtonPermission(binding: DirectiveBinding): boolean {
  const userStore = useUserStore()
  const permissionCode = binding.value
  const mode = binding.arg  // undefined=权限码, 'role'=角色码

  // 空值直接放行
  if (!permissionCode) return true

  if (mode === 'role') {
    // 角色判断模式
    if (Array.isArray(permissionCode)) {
      return userStore.hasAnyRoles(permissionCode)
    }
    return userStore.hasRole(permissionCode)
  }

  // 权限码判断模式（默认）
  if (Array.isArray(permissionCode)) {
    return userStore.hasAnyPermission(permissionCode)
  }
  return userStore.hasPermission(permissionCode)
}

// UserStore 核心判断方法
function hasPermission(code: string): boolean {
  // 第一优先级：超级管理员直接放行
  if (roleCodes.includes('super_admin')) return true
  // 第二优先级：检查权限码列表（含 menuCodes + buttonCodes）
  return permissions.includes(code)
}

// 权限码来源：登录时合并菜单权限码和按钮权限码
permissions = [...data.permissions, ...data.button_codes]
```

### 5.2 后端按钮权限判断

```php
// AdminAuth 核心权限验证
function check(string|array $permission, int $type = 1): bool {
    // 未登录 → 拒绝
    if ($this->userId <= 0) return false;

    // 超级管理员 → 放行
    if ($this->isSuperAdmin()) return true;

    // 按类型检查权限码
    return match($type) {
        1 => in_array($permission, $this->permissions),    // 通用权限
        2 => in_array($permission, $this->getMenuCodes()),  // 菜单权限
        3 => in_array($permission, $this->getApiCodes()),   // API权限
        default => false
    };
}

// 超级管理员判断
function isSuperAdmin(): bool {
    if ($this->userId <= 0) return false;
    $superAdminCode = Config::get('auth.super_admin_code', 'super_admin');
    return in_array($superAdminCode, array_column($this->roles, 'code'));
}

// 按钮权限码获取
function getButtonCodes(): array {
    // 超管：返回全表所有启用按钮码
    if ($this->isSuperAdmin()) return $this->getAllButtonCodes();

    // 非超管：通过角色关联查询
    // 缓存优先 → 数据库查询
    return SimpleCache::remember($cacheKey, $cacheTime, function() {
        $roleIds = array_column($this->roles, 'id');
        $buttonIds = Role::getRoleButtonsByRoleIds($roleIds);
        return Db::name('menu_button')
            ->whereIn('id', $buttonIds)
            ->where('status', 1)
            ->whereNull('delete_time')
            ->column('code');
    });
}
```

### 5.3 API 接口权限判断

```php
// ApiPermission 中间件
function handle(Request $request, Closure $next) {
    $userId = $request->userInfo['id'] ?? 0;

    // 未登录 → 401
    if ($userId <= 0) return json(['code' => 401], 401);

    $auth = AdminAuth::instance();
    $auth->setUser($userId);

    // 超级管理员 → 放行
    if ($auth->isSuperAdmin()) return $next($request);

    // 白名单接口 → 放行
    if ($this->isWhitelisted($method, $path)) return $next($request);

    // 匹配 API 配置
    $api = Api::matchApiByPath($method, $path);

    // 未配置接口 → 默认放行（记录日志）
    if ($api === null) return $next($request);

    // 验证 API 权限码
    if (!in_array($api['code'], $auth->getApiCodes())) {
        return json(['code' => 403, 'msg' => '无权限访问此接口'], 403);
    }

    return $next($request);
}
```

### 5.4 数据权限过滤伪代码

```php
// 在控制器/服务层调用
function applyDataScopeFilter($query, AdminAuth $auth) {
    // 超管不受限制
    if ($auth->isSuperAdmin()) return $query;

    $scopedDeptIds = $auth->getScopedDeptIds();

    switch ($auth->getDataScope()) {
        case 1:  // 全部数据 - 不添加过滤
            break;
        case 2:  // 本部门
        case 3:  // 本部门及以下
        case 5:  // 自定义
            if (!empty($scopedDeptIds)) {
                $query->where('dept_id', 'in', $scopedDeptIds);
            }
            break;
        case 4:  // 仅本人
            $query->where('user_id', '=', $auth->getUserId());
            break;
    }

    return $query;
}
```
