### 1.2 逐表差异对比

#### 1.2.1 sys_user（用户表） — 重大差异

| 对比项 | 迁移文件定义 | 实际数据库 | 差异分析 |
|--------|-------------|-----------|---------|
| `uk_email` 唯一索引 | 001 和 003 均定义 | **不存在** | 实际数据库从未执行 003 迁移；001 中定义的 uk_email 与 003 重复 |
| `uk_mobile` 唯一索引 | 001 和 003 均定义 | **不存在** | 同上 |
| `idx_mobile` 普通索引 | 未定义 | **存在** | 实际数据库手动添加了普通索引，替代了唯一索引 |
| `dept_id` 字段 | 005/006 计划废弃 | **仍存在且大量使用** | 迁移未完成，代码中双写 dept_id 和 sys_user_dept |

**问题详解**：

1. **唯一索引重复定义**：001_init_schema.sql 第 44-45 行已定义 `uk_email` 和 `uk_mobile`，003_add_user_unique_indexes.sql 又重复添加，若按顺序执行会导致 "Duplicate key name" 错误。

2. **唯一索引与软删除冲突**：`uk_email`/`uk_mobile` 不区分软删除记录。当用户被软删除后（delete_time 非空），其 email/mobile 仍占据唯一索引位置，新用户无法使用相同值。应用层校验（UserController）使用 `whereNull('delete_time')` 过滤软删除记录，会通过校验但数据库层会报 `Integrity constraint violation`。

3. **应用层校验不一致**：UserController 使用 `whereNull('delete_time')` 过滤软删除记录检查唯一性，而 ProfileController 使用 `withTrashed()` 包含软删除记录检查唯一性，两者策略矛盾。

## 2. 实施方案

### 2.1 方案选择：方案二 — 基于实际数据库进行架构重构

**选择理由**：

1. 实际数据库是当前运行的"真实状态"，代码也是基于实际数据库编写的
2. 迁移文件存在多处不一致（重复定义、未执行的迁移、缺失的表）
3. 直接基于实际数据库重构可以确保与现有代码完全兼容
4. 整合后的迁移文件作为"单一真相来源"，未来新部署时只需执行两个文件

### 3.2 关键设计变更

#### 3.2.1 sys_user 索引优化

**原方案问题**：
```sql
UNIQUE KEY `uk_email` (`email`),    -- 不区分软删除，软删除后无法复用
UNIQUE KEY `uk_mobile` (`mobile`),  -- 同上
```

**新方案**：
```sql
KEY `idx_mobile` (`mobile`),                    -- 普通索引，加速查询
KEY `idx_email_active` (`email`, `delete_time`), -- 复合索引，支持"活跃用户唯一性"查询
KEY `idx_mobile_active` (`mobile`, `delete_time`) -- 复合索引，同上
```

**设计考量**：
- MySQL 的 UNIQUE 约束对 NULL 值不生效（NULL != NULL），但 email/mobile 为空字符串时仍会冲突
- 软删除场景下，唯一索引会阻止新用户使用已删除用户的 email/mobile
- 改用复合索引 + 应用层校验（已有）的方案，兼顾查询性能和数据灵活性
- 应用层校验需统一使用 `whereNull('delete_time')` 策略（见维护建议）

#### 3.2.2 sys_user.dept_id 双写策略

**现状**：
- `sys_user.dept_id`：存储主部门ID（冗余字段）
- `sys_user_dept`：存储所有部门关联（多对多）
- 代码通过 `syncUserDepts()` 等方法维护两者同步

**设计决策**：保留双写策略，原因如下：
1. `dept_id` 在代码中被大量使用（用户列表过滤、数据权限判断、导出等）
2. 移除 `dept_id` 需要改动 20+ 处代码，风险高
3. 双写保证了查询性能（避免 JOIN sys_user_dept）
4. 在 001 中明确注释 `dept_id` 为冗余字段

#### 3.2.3 wx_user 移除 delete_time

**原因**：
- WxUser 模型继承 BaseModel 但实际未启用软删除
- 实际数据库中不存在 delete_time 字段
- 微信用户不存在"软删除"业务场景（注销即删除）

## 4. 数据迁移策略

### 4.1 现有数据库升级路径

对于已运行的 rbac_system 数据库，需执行以下增量迁移：

```sql
-- 步骤1：修复 sys_user 索引
ALTER TABLE `sys_user` DROP INDEX IF EXISTS `uk_email`;
ALTER TABLE `sys_user` DROP INDEX IF EXISTS `uk_mobile`;
ALTER TABLE `sys_user` ADD KEY `idx_email_active` (`email`, `delete_time`);
ALTER TABLE `sys_user` ADD KEY `idx_mobile_active` (`mobile`, `delete_time`);

-- 步骤2：执行 006 数据迁移（将 dept_id 迁移到 sys_user_dept）
INSERT INTO `sys_user_dept` (`user_id`, `dept_id`, `is_primary`, `sort`, `create_time`)
SELECT `id`, `dept_id`, 1, 0, NOW()
FROM `sys_user`
WHERE `dept_id` IS NOT NULL AND `dept_id` > 0
AND NOT EXISTS (
  SELECT 1 FROM `sys_user_dept`
  WHERE `sys_user_dept`.`user_id` = `sys_user`.`id`
  AND `sys_user_dept`.`dept_id` = `sys_user`.`dept_id`
);

-- 步骤3：创建缺失的 business 表
CREATE TABLE IF NOT EXISTS `business` (
  `id` BIGINT UNSIGNED NOT NULL AUTO_INCREMENT COMMENT '主键ID',
  `title` VARCHAR(200) NOT NULL COMMENT '标题',
  `category` VARCHAR(50) DEFAULT NULL COMMENT '分类',
  `cover` VARCHAR(500) DEFAULT NULL COMMENT '封面图URL',
  `content` LONGTEXT DEFAULT NULL COMMENT '内容',
  `summary` VARCHAR(500) DEFAULT NULL COMMENT '摘要',
  `status` TINYINT NOT NULL DEFAULT 1 COMMENT '状态：0禁用 1正常',
  `sort` INT DEFAULT 0 COMMENT '排序',
  `created_by` BIGINT UNSIGNED DEFAULT NULL COMMENT '创建者',
  `create_time` DATETIME DEFAULT NULL COMMENT '创建时间',
  `updated_by` BIGINT UNSIGNED DEFAULT NULL COMMENT '更新者',
  `update_time` DATETIME DEFAULT NULL COMMENT '更新时间',
  `delete_time` DATETIME DEFAULT NULL COMMENT '删除时间',
  PRIMARY KEY (`id`),
  KEY `idx_category` (`category`),
  KEY `idx_status` (`status`),
  KEY `idx_sort` (`sort`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='商务内容表';

-- 步骤4：创建缺失的 business_interaction 表
CREATE TABLE IF NOT EXISTS `business_interaction` (
  `id` BIGINT UNSIGNED NOT NULL AUTO_INCREMENT COMMENT '主键ID',
  `wx_user_id` BIGINT UNSIGNED NOT NULL COMMENT '微信用户ID',
  `business_id` BIGINT UNSIGNED NOT NULL COMMENT '商务内容ID',
  `type` VARCHAR(20) NOT NULL COMMENT '交互类型：favorite/like/collect',
  `create_time` DATETIME DEFAULT NULL COMMENT '创建时间',
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_user_business_type` (`wx_user_id`, `business_id`, `type`),
  KEY `idx_business_id` (`business_id`),
  KEY `idx_type` (`type`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='用户交互表';
```

### 4.2 全新部署路径

对于新环境，只需执行整合后的两个文件：
1. `001_init_schema.sql` — 创建全部 20 张表
2. `002_init_data.sql` — 初始化数据 + 部门关联迁移

### 4.3 数据完整性保障

| 检查项 | 保障措施 |
|--------|---------|
| 部门关联数据 | 006 迁移使用 `NOT EXISTS` 防止重复插入 |
| 唯一索引安全 | 先 DROP IF EXISTS 再 ADD，避免重复执行报错 |
| 外键约束 | 迁移期间关闭 `FOREIGN_KEY_CHECKS`，完成后恢复 |
| 业务数据 | business/business_interaction 使用 `CREATE IF NOT EXISTS`，幂等执行 |

## 5. 维护建议

### 5.1 短期（立即执行）

1. **统一应用层唯一性校验策略**：将 ProfileController 中的 `withTrashed()` 改为 `whereNull('delete_time')`，与 UserController 保持一致。

2. **执行现有数据库增量迁移**：按 4.1 节的步骤执行，修复索引、迁移部门数据、创建缺失表。

3. **清理临时文件**：删除 `database/check_db.php`（调试用脚本，不应保留在项目中）。

### 5.2 中期（1-2 周内）

1. **引入迁移版本管理工具**：建议使用 ThinkPHP 的 `topthink/think-migration` 扩展（基于 Phinx），替代手动 SQL 文件管理。优势：
   - 版本化迁移，支持 up/down 回滚
   - 迁移状态追踪（phinxlog 表）
   - 团队协作时避免迁移冲突

2. **完善 business 表字段**：当前 business 表字段基于模型推断，可能遗漏实际业务需要的字段（如 author、source、tags 等）。建议与前端团队确认完整字段需求后更新。

3. **添加数据库外键约束**：当前所有关联表仅通过应用层维护数据一致性，建议在核心关联表（sys_user_role、sys_user_dept、sys_role_menu 等）上添加外键约束，从数据库层面保障引用完整性。

### 5.3 长期（架构优化）

1. **逐步消除 dept_id 双写**：长期目标是移除 `sys_user.dept_id`，完全依赖 `sys_user_dept` 管理用户部门关系。实施步骤：
   - 第一阶段（当前）：双写 + dept_id 为主查询字段
   - 第二阶段：所有查询改用 sys_user_dept，dept_id 仅作为缓存
   - 第三阶段：移除 dept_id 字段

2. **考虑分表策略**：sys_login_log 和 sys_operation_log 会持续增长，建议：
   - 按月/季度分区（PARTITION BY RANGE）
   - 或定期归档到历史表

3. **统一模型基类**：WxConfig 直接继承 `think\Model` 而非 `BaseModel`，建议统一继承 BaseModel 并在模型中按需关闭软删除，保持代码一致性。
